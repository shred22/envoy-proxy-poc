# Envoy mTLS Sidecar — Proof of Concept

A Spring Boot application paired with an Envoy sidecar proxy that offloads **all mTLS concerns** from the application. The app communicates over plain HTTP/gRPC on loopback; Envoy handles certificate management, TLS termination, and TLS origination on every boundary.

---

## Architecture

```
                        ┌─────────────────────────────────────────┐
                        │              Pod / Container             │
                        │                                          │
External Client         │   ┌──────────┐        ┌──────────────┐  │
  (mTLS HTTPS)  ──────▶ │   │  Envoy   │──HTTP──▶ Spring Boot  │  │
                        │   │ :8443    │◀──HTTP──│    :8080     │  │
                        │   │          │        └──────────────┘  │
                        │   │  Egress  │                          │
Spring Boot             │   │  gRPC    │──mTLS──▶ External gRPC   │
  localhost:9090 ──────▶│   │  :9090   │          Service :443    │
                        │   │          │                          │
Spring Boot             │   │  Egress  │                          │
  localhost:9091 ──────▶│   │  HTTP    │──mTLS──▶ External REST   │
                        │   │  :9091   │          API :443        │
                        │   └──────────┘                          │
                        └─────────────────────────────────────────┘
```

### Traffic flows

| Direction | From | To | TLS |
|---|---|---|---|
| Ingress | External caller | Envoy `:8443` → Spring Boot `:8080` | Envoy terminates mTLS, forwards plain HTTP |
| Egress gRPC | Spring Boot `:9090` | Envoy → External gRPC `:443` | Envoy originates mTLS |
| Egress HTTP | Spring Boot `:9091` | Envoy → External REST `:443` | Envoy originates mTLS |

### Port reference

| Port | Owner | Purpose |
|---|---|---|
| `8080` | Spring Boot | HTTP (loopback + local test only) |
| `6565` | Spring Boot | gRPC server (loopback only) |
| `8443` | Envoy | Ingress — mTLS from external callers |
| `9090` | Envoy | Egress — gRPC listener (loopback) |
| `9091` | Envoy | Egress — HTTP listener (loopback) |
| `9901` | Envoy | Admin API |

## Quick Start — Local (Docker Compose)

### 1. Clone and generate certificates

```bash
git clone <repo-url>
cd envoy-POC
bash scripts/gen-certs.sh
```

This creates `certs/` with:

```
certs/
├── ca.crt / ca.key          # Self-signed CA
├── tls.crt / tls.key        # Server cert (Envoy ingress)
└── client.crt / client.key  # Client cert (for mTLS callers)
```

### 2. Run the full stack

```bash
docker compose up --build
```

Services started:

| Service | Host port(s) |
|---|---|
| Envoy | 8080, 8443, 9090, 9091, 9901 |
| grpc-echo mock | 50051 |
| rest-mock (WireMock) | 8090 |

### 3. Run end-to-end tests

```bash
bash scripts/local-test.sh
```

The script:
1. Generates certs if missing
2. Builds and starts all containers
3. Waits for Spring Boot to be healthy (`/actuator/health/readiness`)
4. Runs 10 test scenarios covering ingress mTLS, gRPC egress, and HTTP egress
5. Prints a pass/fail summary

### 4. Manual smoke tests

```bash
# Ingress — mTLS ping
curl --cacert certs/ca.crt \
     --cert certs/client.crt \
     --key  certs/client.key \
     https://localhost:8443/api/ping

# Ingress — gRPC egress (via Spring Boot controller)
curl --cacert certs/ca.crt \
     --cert certs/client.crt \
     --key  certs/client.key \
     https://localhost:8443/api/identity/user-42

# Ingress — REST egress
curl --cacert certs/ca.crt \
     --cert certs/client.crt \
     --key  certs/client.key \
     https://localhost:8443/api/external/resource/res-99

# Direct gRPC (bypasses Envoy, talks to grpc-echo mock)
grpcurl -plaintext \
        -d '{"subject_id":"test-user"}' \
        localhost:50051 \
        identity.IdentityService/GetIdentity

# Envoy admin
curl http://localhost:9901/ready
curl http://localhost:9901/clusters
curl "http://localhost:9901/stats?filter=circuit_breakers"
```

## Spring Boot Application

### Endpoints

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/ping` | Ingress health check — echoes XFCC peer cert header |
| `GET` | `/api/identity/{subjectId}` | Calls external gRPC identity service via Envoy egress |
| `POST` | `/api/echo` | Sends echo over gRPC via Envoy egress |
| `GET` | `/api/external/resource/{id}` | Calls external REST API via Envoy egress |
| `POST` | `/api/external/resource` | Creates resource on external REST API via Envoy egress |
| `GET` | `/actuator/health` | Spring Boot health (liveness + readiness) |

### Key configuration (`application.yaml`)

```yaml
server:
  port: 8080                          # plain HTTP — Envoy handles TLS

grpc:
  client:
    identity-service:
      address: static://127.0.0.1:9090   # Envoy egress gRPC listener
      negotiation-type: plaintext         # no TLS — Envoy handles it

egress:
  rest:
    base-url: http://127.0.0.1:9091      # Envoy egress HTTP listener
```

### gRPC proto contract

```protobuf
service IdentityService {
  rpc GetIdentity (IdentityRequest) returns (IdentityResponse);
  rpc Echo        (EchoRequest)     returns (EchoResponse);
}
```

## OpenShift Deployment

### Prerequisites

- `oc` CLI authenticated to your cluster
- The app container image built and pushed to your registry
- Certificates stored in an OCP Secret (see below)

### 1. Create the TLS Secret

```bash
# gen-certs.sh already wrote openshift/tls-secret.yaml
oc apply -f openshift/tls-secret.yaml -n envoy-poc
```

Or from real certs:

```bash
oc create secret generic envoy-mtls-certs \
  --from-file=tls.crt=/path/to/server.crt \
  --from-file=tls.key=/path/to/server.key \
  --from-file=ca.crt=/path/to/ca.crt \
  -n envoy-poc
```

### 2. Apply SCC binding

```bash
oc apply -f openshift/scc-binding.yaml -n envoy-poc
```

### 3. Process and apply the ConfigMap template

```bash
oc process -f openshift/configmap-template.yaml \
  -p BRANCH_NAME_HY=main \
  -p PROJECT_NAME=envoy-poc \
  -p GRPC_SERVICE_HOST=your-grpc-service.example.com \
  -p REST_SERVICE_HOST=your-rest-api.example.com \
  | oc apply -n envoy-poc -f -
```

### 4. Process and apply the Deployment template

```bash
oc process -f openshift/deployment-template.yaml \
  -p BRANCH_NAME_HY=main \
  -p PROJECT_NAME=envoy-poc \
  -p APP_IMAGE=image-registry.openshift-image-registry.svc:5000/envoy-poc/app \
  -p IMAGE_TAG=latest \
  -p ROUTE_HOST=envoy-poc.apps.your-cluster.example.com \
  | oc apply -n envoy-poc -f -
```

### OpenShift resources created

| Resource | Name | Purpose |
|---|---|---|
| `ConfigMap` | `envoy-poc-<branch>-envoy` | Envoy bootstrap + SDS files |
| `Deployment` | `envoy-poc-<branch>` | Pod with app + envoy sidecar |
| `Service` | `envoy-poc-<branch>` | ClusterIP on port 8443 |
| `Route` | `envoy-poc-<branch>` | TLS passthrough (Envoy handles mTLS) |
| `ServiceAccount` | `envoy-poc-sa` | Identity for the pod |
| `RoleBinding` | `envoy-poc-sa-scc` | Grants `restricted-v2` SCC |

### OCP Security notes

All containers run as non-root (`runAsNonRoot: true`, UID 1001). All ports are above 1024. The `restricted-v2` SCC is sufficient — no `privileged` or `anyuid` required.

See [ENVOY-CONFIG.md](ENVOY-CONFIG.md) for a detailed breakdown of the Envoy configuration.

## Certificate File Layout

```
/etc/envoy/
├── config/
│   └── envoy.yaml                        # Mounted from ConfigMap (envoy.yaml key)
├── sds/                                  # Mounted from ConfigMap (separate keys)
│   │
│   │   ── OCP (configmap-template.yaml) ──
│   ├── server_cert_sds.yaml              # server_cert secret  (ingress listener)
│   ├── client_cert_sds.yaml              # client_cert secret  (egress clusters)
│   ├── validation_context_sds_secret.yaml# CA bundle           (ingress + egress)
│   │
│   │   ── Local Docker (envoy-local.yaml) ──
│   └── tls_certificate_sds_secret.yaml   # server_cert only    (one secret per file)
│
└── certs/                                # Mounted from OCP Secret / local certs/
    ├── tls.crt                           # Server certificate (+ chain)
    ├── tls.key                           # Server private key
    └── ca.crt                            # CA bundle for peer validation
```

> **Why two separate SDS files on OCP?**
> Envoy's path-based SDS enforces exactly one secret per file. The ingress listener needs `server_cert` and the egress clusters need `client_cert` — they must live in separate files (`server_cert_sds.yaml` and `client_cert_sds.yaml`). The local Docker config only uses `server_cert` (egress is plaintext), so one file is sufficient there.
